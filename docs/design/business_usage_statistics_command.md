# 业务使用深度与活跃度统计脚本使用说明

## 概述

`business_usage_statistics` 是一个Django management command脚本，用于统计各业务的标准运维使用深度与活跃度指标，并生成加权综合评分。

## 功能特性

### 统计指标

#### 基础信息
- **业务ID**: 业务的唯一标识
- **业务名称**: 业务的显示名称
- **业务沟通负责人**: 业务执行者（executor）或开发商账号（cc_owner）

#### 使用深度指标
1. **标准运维模版数量**: 该业务下所有项目中的标准运维模板总数
2. **周期任务数量**: 该业务下所有项目中的周期任务总数
3. **计划任务数量**: 该业务下所有项目中的计划任务总数
4. **轻应用数量**: 该业务下所有项目中的轻应用总数
5. **标准插件使用多样性**: 该业务使用的标准插件种类数量
6. **子流程使用率**: 使用子流程的模板占比

#### 活跃度指标
1. **最近N天执行次数**: 指定时间范围内的任务执行总次数（默认365天）
2. **总执行次数**: 所有历史任务执行总次数
3. **活跃用户数**: 指定时间范围内创建或执行任务的用户数量（去重）
4. **执行频率**: 平均每天执行次数

#### 质量指标
1. **任务成功率**: 指定时间范围内任务执行的成功率（百分比）
2. **模板复用率**: 被使用的模板占所有模板的比例（百分比）
3. **平均执行时长**: 指定时间范围内任务的平均执行时长（秒）

#### 综合评分
- **加权综合评分**: 基于所有指标的加权综合评分（0-100分）

## 使用方法

### 基本用法

```bash
# 输出到标准输出（默认统计最近365天）
python manage.py business_usage_statistics

# 输出到CSV文件
python manage.py business_usage_statistics --output /path/to/output.csv

# 指定统计时间范围（最近180天）
python manage.py business_usage_statistics --execution-days 180 --output /path/to/output.csv

# 包含已停用的业务
python manage.py business_usage_statistics --include-inactive --output /path/to/output.csv
```

### 参数说明

- `--output`: 输出CSV文件路径（可选），如果不指定则输出到标准输出
- `--execution-days`: 统计执行次数的时间范围（天数），默认365天
- `--include-inactive`: 是否包含已停用的业务，默认不包含

### 输出格式

输出为CSV格式，包含以下列：

| 列名 | 说明 |

| 业务沟通负责人 | 业务执行者或开发商账号 |
| 项目数量 | 该业务下的项目总数 |
| 标准运维模版数量 | 标准运维模板总数 |
| 周期任务数量 | 周期任务总数 |
| 计划任务数量 | 计划任务总数 |
| 轻应用数量 | 轻应用总数 |
| 最近N天执行次数 | 指定时间范围内的执行次数 |

| 活跃用户数 | 活跃用户数量 |

## 加权评分算法

加权综合评分基于以下权重配置：

### 使用深度指标（50%）
- 模板数量: 15%
- 周期任务数量: 10%
- 计划任务数量: 5%
- 轻应用数量: 5%
- 标准插件使用多样性: 10%
- 子流程使用深度: 5%

### 活跃度指标（40%）
- 最近N天执行次数: 20%
- 活跃用户数: 10%
- 执行频率: 10%

### 质量指标（10%）
- 任务成功率: 5%
- 模板复用率: 5%

### 评分计算方式

1. **归一化处理**: 对于数量类指标，使用最大值归一化（0-100分）
2. **百分比指标**: 直接使用百分比值（0-100分）
3. **加权求和**: 各指标得分乘以对应权重后求和

```bash
    --execution-days 180 \
```

### 示例3：包含所有业务（包括已停用）

```bash

    --include-inactive \
    --output /tmp/business_statistics_all.csv
```

## 数据源说明

所有指标均基于以下数据表：

- `gcloud_core_business` - 业务表
- `gcloud_core_project` - 项目表
- `tasktmpl3_tasktemplate` - 标准运维模板表
- `periodictask_periodictask` - 周期任务表
- `clocked_task_clockedtask` - 计划任务表
- `gcloud_contrib_appmaker_appmaker` - 轻应用表
- `taskflow3_taskflowinstance` - 任务实例表
- `pipeline_pipeline_instance` - 流程实例表
- `gcloud_analysis_statistics_taskflowstatistics` - 任务流统计表
- `gcloud_analysis_statistics_templatestatistics` - 模板统计表
- `gcloud_analysis_statistics_templatenodestatistics` - 模板节点统计表

## 注意事项

1. **业务与项目的关系**: 一个业务可以对应多个项目，通过 `Project.bk_biz_id` 关联到 `Business.cc_id`

2. **数据过滤**:
   - 默认只统计未停用的项目（`is_disable=False`）
   - 默认只统计未删除的模板和轻应用（`is_deleted=False`）
   - 默认只统计未归档的业务（`status != 'disabled'`）

3. **时间范围**: 执行次数统计默认使用最近365天，可通过 `execution_days` 参数调整

4. **业务沟通负责人**: 优先使用 `Business.executor`，如果为空则使用 `Business.cc_owner`

5. **性能考虑**: 对于大量业务的统计，建议：
   - 添加适当的数据库索引
   - 在非高峰期运行
   - 使用 `--output` 参数将结果保存到文件

## 应用场景

### 运营分析
- 识别高活跃度业务
- 发现僵尸业务（没有使用记录）
- 评估各业务的使用深度

### 资源规划
- 根据使用情况分配资源
- 识别需要重点支持的业务
- 制定业务推广策略

### 业务健康度评估
- 综合评估业务的使用情况
- 识别潜在问题业务
- 制定优化方案

## 故障排查

### 问题1：没有找到符合条件的业务

**原因**: 可能所有业务都已停用，或者数据库中没有业务数据

**解决**: 使用 `--include-inactive` 参数包含已停用的业务

### 问题2：执行时间过长

**原因**: 业务数量较多或数据量较大

**解决**:
- 在非高峰期运行
- 考虑添加数据库索引
- 使用 `--execution-days` 参数缩小统计时间范围

### 问题3：某些业务指标为0

**原因**: 该业务下没有项目，或项目下没有相关数据

**解决**: 这是正常现象，表示该业务确实没有使用相关功能
