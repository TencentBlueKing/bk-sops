# 职能化任务创建时参数值丢失问题分析

## 一、问题描述

用户创建"职能化任务"时，流程入参 `a` 传入的是 `a_value`，但：
1. 实际流程执行时，参数 `a` 使用的是模板默认值 `none`（字符串）
2. 任务操作流水记录的 `extra_info` 中 `constants` 里 `a` 的值也为 `none`

**已知条件：**
- 变量 `show_type` = `show`（可见变量）
- 变量 `custom_type` = `input`（文本输入类型）
- 模板默认值为字符串 `"none"`
- 该问题为首次发现，之前执行正常

## 三、后端代码分析（无 Bug）

### 3.1 `perform_create` 方法

**文件：** `gcloud/core/apis/drf/viewsets/taskflow.py` 第 442-495 行

def perform_create(self, serializer):
    template = serializer.validated_data.pop("template")
    pipeline_instance_kwargs = {
        "pipeline_tree": serializer.validated_data.pop("pipeline_tree"),

    pipeline_instance = TaskFlowInstance.objects.create_pipeline_instance(

### 3.2 `create_pipeline_instance` 方法

**文件：** `gcloud/taskflow3/models.py` 第 572-608 行

六步变换，**全部不涉及 constants 值修改**：

| # | 函数 | 修改了什么 | 是否修改顶层 constants |
|---|------|-----------|----------------------|
| 1 | `replace_template_id()` | SubProcess 节点的 template_id | **否** |

### 3.3 `extract_extra_info` 函数

**文件：** `gcloud/contrib/operate_record/utils.py` 第 17-23 行

```python
def extract_extra_info(constants, keys=None):
    extra_info = {}
    if not constants:
        return ""
    for key in list(constants.keys()) if not keys else keys:
        extra_info.update({
            key: {"name": constants[key]["name"], "value": constants[key]["value"]}
        })
    return json.dumps(extra_info, ensure_ascii=False)
```

纯读取操作，不修改任何数据。

### 3.4 后端结论

> **后端从接收 `pipeline_tree` 到生成 `extra_info` 的完整链路中，没有任何环节会修改 constants 中变量的 `value` 值。操作记录中 `extra_info` 的 constants 内容完全取决于前端提交的 `pipeline_tree` 中的 constants。**

#### 4.2.2 `constants` watcher — 潜在风险点

```javascript
watch: {
    constants (val) {
        this.variables = tools.deepClone(val)
        this.getFormData()  // ⚠️ 重新初始化，会重置 renderData！
    }
}
```

**风险**：如果 `constants` prop 引用发生变化，会触发 `getFormData()`，将 `renderData` 重置为模板默认值，**丢弃所有用户已填写的参数值**。

#### 4.2.3 `getVariableData()` — 收集表单数据

**文件：** 第 334-375 行

```javascript

        if (variable.show_type === 'hide') {
            // 隐藏变量处理...
        } else {
            variable.value = this.renderData[key]      // ← 用 renderData 覆盖

    return Promise.resolve(variables)
}
```

关键赋值：`variable.value = this.renderData[key]`
- 如果 `renderData[key]` = `"a_value"` → 正确
- 如果 `renderData[key]` = `"none"` → 用户输入丢失，使用了模板默认值

## 五、问题定位与根因分析

### 5.1 排除项

| 排查项 | 结论 | 依据 |
|--------|------|------|
| 后端 perform_create 逻辑 | ✅ 无 Bug | 6 步变换均不触碰 constants 值 |
| 后端 extract_extra_info | ✅ 无 Bug | 纯读取操作 |
| 后端 Serializer 反序列化 | ✅ 无 Bug | json.loads + 校验，不修改数据 |
| 后端 FunctionTask 创建 | ✅ 无 Bug | 仅创建关联记录 |
| 后端 execution_snapshot 存储 | ✅ 无 Bug | 原样存储 pipeline_tree |
| 职能化任务认领(task_claim) | ✅ 不影响 | 认领发生在操作记录创建之后 |

### 5.2 关键发现

**操作记录中 `extra_info` 的值 = 模板默认值 `"none"`，而非用户输入的 `"a_value"`。**

由于后端不修改 constants，这意味着：**前端提交的 `pipeline_tree` 中，变量 `a` 的 value 就已经是 `"none"`**。

### 5.3 前端可能的原因

#### 原因1：`renderData[key]` 未被用户输入更新（最可能）

对于 `custom_type = 'input'` 的变量，RenderForm 通过 `v-model` 绑定 `renderData`。正常情况下用户输入会触发更新。但以下边界场景可能导致更新失败：

- **表单组件 v-model 事件未触发**：例如浏览器自动填充（autofill）、程序化赋值、粘贴后未触发 `input/change` 事件
- **RenderForm 内部 `$emit('input')` 时机问题**：某些浏览器或输入法（如中文输入法组合输入中）可能导致 `input` 事件被延迟或丢失

#### 原因2：`constants` watcher 意外触发（可能性较低）

```javascript
watch: {
    constants (val) {
        this.variables = tools.deepClone(val)
        this.getFormData()   // 重置 renderData 为模板默认值
    }
}
```

如果 `pipelineData.constants` 对象引用在用户填参后发生了变化（例如 Vue 响应式系统的内部替换），会导致 `getFormData()` 被重新调用，**将 `renderData` 全部重置为模板默认值**。

在当前代码中，`pipelineData` 在 `loadData()` 中被赋值后不会被再次修改，因此此 watcher 正常情况下不会再次触发。但以下情况可能例外：
- Vue 2 响应式系统在极端情况下对对象引用的重新代理
- 父组件重新渲染导致 prop 引用变化

#### 原因3：用户操作未实际生效（无法排除）

用户可能在以下场景下误以为已填写参数：
- 输入后未移出焦点（某些组件需要 blur 事件才触发 v-model 更新）
- 输入了值但随后被浏览器插件或快捷键撤销
- 多个同名参数混淆

## 六、最终结论

### 是否存在代码 Bug？

**后端代码：确认无 Bug。** 从 `perform_create` → `create_pipeline_instance` → `extract_extra_info` 的完整链路中，没有任何逻辑会修改 `constants` 的 `value` 值。操作记录的 `extra_info` 忠实反映了 `execution_snapshot` 中存储的数据。

**前端代码：未发现系统性 Bug，但存在以下潜在风险：**

1. **`getVariableData()` 缺乏防御性编码**（第 368 行）：
   ```javascript
   variable.value = this.renderData[key]
   ```
   如果 `renderData[key]` 为 `undefined`（即该 key 不存在于 renderData 中），`value` 会被设为 `undefined`，JSON 序列化时该字段会丢失。建议改为：
   ```javascript
   variable.value = (key in this.renderData) ? this.renderData[key] : variable.value
   ```

2. **`constants` watcher 存在重置风险**（第 98-102 行）：
   ```javascript
   watch: {
       constants (val) {
           this.variables = tools.deepClone(val)
           this.getFormData()  // 会完全重置 renderData
       }
   }
   ```
   该 watcher 一旦触发，会丢弃所有用户已填写的参数。建议增加保护机制，避免在用户已填参的情况下重置表单。

### 综合判断

本次问题**最可能**是由前端参数收集环节的边界场景导致：用户填入的值 `"a_value"` 未能成功写入 `renderData`（或写入后被意外重置），导致提交给后端的 `pipeline_tree` 中该变量仍为模板默认值 `"none"`。

鉴于该问题为首次发现且无法稳定复现，建议：

1. **增加前端日志**：在 `getVariableData()` 中增加日志，记录 `renderData` 的完整内容与最终提交的 constants，便于后续排查
2. **增加后端日志**：在 `perform_create` 中记录接收到的 `pipeline_tree` 中 constants 的值
3. **加固 `getVariableData()`**：增加防御性编码，当 `renderData[key]` 不存在时保留原始值
4. **用户侧复现验证**：请用户在浏览器 DevTools Network 面板中抓取创建任务请求，确认实际发送的 `pipeline_tree` 中该参数的值
